🧪 Test Cases (QA Scenario)
1. UI/UX & Navigation Test Cases
TC-01: Edit Form Opens Successfully

Step: Click on any transaction from the list.

Expected Result: The Edit Form opens successfully as a modal or a new page. It contains all 3 tabs: "Detailed Information", "Images", and "Transaction History". The "Detailed Information" tab is selected by default.

TC-02: Tab Switching

Step: Click through each tab sequentially ("Detailed Information" $\rightarrow$ "Images" $\rightarrow$ "Transaction History").

Expected Result: Tab switching is smooth. Any unsaved data currently being inputted in other tabs remains intact (unless the "Cancel" button is clicked).

2. Tab 1: Detailed Information (Amount Lock Logic)
TC-03: New Transaction (< 15 mins) — Amount is Editable

Step: Open a transaction that was created 5 minutes ago $\rightarrow$ Check the "Amount" field.

Expected Result: The "Amount" field is Editable. The user can successfully change the value and save the transaction.

TC-04: Old Transaction (> 15 mins) — Amount is Locked

Step: Open a transaction that was created more than 16 minutes ago (or days ago) $\rightarrow$ Check the "Amount" field.

Expected Result: The "Amount" field is Disabled / Read-only. The cursor cannot focus on it, and the user cannot edit the value. (Optional: A lock icon or tooltip appears explaining the 15-minute rule).

TC-05: Time Boundary Test (Exactly 15 mins)

Step: Find or simulate a transaction created exactly 15 minutes and 00 seconds ago $\rightarrow$ Open the edit form.

Expected Result: The system handles the edge case correctly based on backend business rules (typically $\ge$ 15 minutes triggers the lock). All other fields remain editable.

3. Tab 2: Images (Max 10 Images Limit)
TC-06: Display Existing Images

Step: Open a transaction that already has 3 images attached.

Expected Result: All 3 images are rendered correctly with the exact same styling and layout as the legacy form.

TC-07: Upload New Images Within Limit

Step: Open a transaction with 3 existing images $\rightarrow$ Upload 2 more images.

Expected Result: Upload succeeds smoothly. Total images count updates to 5.

TC-08: Exceeding Maximum Limit (Max = 10 images)

Step: Open a transaction with 9 existing images $\rightarrow$ Try uploading 2 more images simultaneously.

Expected Result: System blocks the upload and triggers a validation error/toast message stating that a maximum of 10 images is allowed. Alternatively, the "Upload" button becomes disabled once the count hits 10.

4. Tab 3: Transaction History
TC-09: Accurate History Log Rendering

Step: Edit a few fields on the current transaction $\rightarrow$ Click Save $\rightarrow$ Re-open the transaction and go to the "Transaction History" tab.

Expected Result: The updated logs are displayed accurately (showing who made the change, which field was modified, Old Value $\rightarrow$ New Value, and Timestamp), copying the exact rendering logic from the legacy component.
