Transform the Halo Ritual feature in the Kenfinly application from a traditional dual-button check-in/check-out system to a streamlined "Invisible Design" model. Implement a single-button interface with automatic countdown logic based on an 8-hour standard timeframe, a 12:00 PM threshold, and timezone-aware auto-close functionality to support millions of daily active users.


📝 Description

Restructure the entire Halo Ritual feature to eliminate manual check-out processes and simplify the codebase. The new design centers on one action button, automatic time tracking, and intelligent session closure based on each user's national timezone.

Your goal is to remove legacy check-out flows completely and optimize the system for scale. This redesign will support daily operations for millions of users while maintaining accuracy and performance.

🎯 Functional Requirements

1. Single-Button Minimalist Interface


Replace the existing rectangular check-in/check-out buttons with one circular button labeled "Halo" positioned at the center of the progress ring.

Remove these elements:

Check-out button
"Server Error" notifications
Explanatory text below the timer (e.g., "Countdown time pool", "08:00 Standard")

Keep these elements:

Circular Halo button
Progress ring
Countdown timer in HH:MM:SS format
2. Countdown Time Calculation Logic (Backend)
Use these parameters to determine countdown duration:

Standard commitment timeframe: 8 hours (28,800 seconds)

Threshold for session classification: 12:00 PM (user's local time)

Calculation rules:

Before or at 12:00 PM: Grant full 8-hour countdown. Progress ring reaches 100% when complete.
After 12:00 PM: Cap maximum time at 4 hours (14,400 seconds). Actual runtime is the minimum value between 4 hours and the seconds remaining until midnight (24:00) on that day.
3. Timezone-Aware Auto-Close Mechanism
Automatically close the work session when the clock reaches midnight (24:00) according to the user's national timezone identifier (e.g., Asia/Ho_Chi_Minh for Vietnam, Asia/Singapore for Singapore).

The system should handle timezone transitions seamlessly without requiring manual intervention from users.

🏗️ Technical Architecture & Database Design
1. Frontend Decoupling (Reduce API Load)
When a user presses the button, the backend returns only two values:

start_time (Unix timestamp)
duration (seconds allocated)
The frontend manages the countdown independently using JavaScript (setInterval) and renders the progress ring without making additional API calls each second.

2. Two-Layer Storage Model (Built for Millions of DAU)
Instant Layer (RAM):
Store active Halo session state in Redis Hash/String with key format: user:halo:status:{user_id}

Set Time-To-Live (TTL) to expire at midnight (24:00) based on the user's timezone.

Persistent Layer (Disk):
Use asynchronous processing to calculate actual seconds achieved and perform bulk inserts into MySQL.


Two implementation options:

Background worker listening to Redis Keyspace Notifications for key expiration events
Hourly cronjob scanning countries that have completed their day
